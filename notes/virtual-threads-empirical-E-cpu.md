# 실증 노트 E: CPU-bound 작업에서 가상 스레드

> **실험일**: 2026-06-18  
> **결론**: CPU-bound 작업에서 가상 스레드와 플랫폼 스레드의 처리량 차이는 미미하다. 병렬도는 코어 수가 상한이며 스레드 종류와 무관하다.

---

## 측정 결과

| threads | Platform(ms) | Virtual(ms) | 비율(V/P) | 해석 |
|---------|-------------|------------|---------|------|
| 6 (cores/2) | 74 | 76 | 1.03 | 동등 |
| **12 (=cores)** | **53** | **58** | **1.09** | 최적 구간 진입 |
| 24 (2×cores) | 54 | 57 | 1.06 | **plateau 진입, 개선 없음** |
| 48 (4×cores) | 51 | 51 | 1.00 | 동등, 완전 plateau |
| 96 (8×cores) | 63 | 55 | **0.87** | 가상이 오히려 근소하게 빠름 |

---

## 가설 검증

### E-1: CPU-bound에서 스레드 수를 늘려도 처리량이 증가하지 않는다

**✅ 확인**

threads=12 → threads=96: 플랫폼 53→63ms(오히려 증가), 가상 58→55ms(거의 동일).  
처리량의 상한 = core 수. 스레드 모델과 무관.

---

## 흥미로운 발견

### threads=96에서 가상 스레드가 0.87x (빠름)

예상과 반대. 이유: 96개 플랫폼 스레드는 OS 스케줄러가 직접 관리 → 96 OS 스레드 간 컨텍스트 스위칭 비용 증가(63ms). 가상 스레드는 ForkJoinPool(12 carriers)이 관리 → 12 OS 스레드만 컨텍스트 스위칭, 나머지는 JVM 레벨에서 스케줄링(55ms).

**주의**: 63ms vs 55ms 차이(8ms)는 단일 실행 기준이며 측정 노이즈 범위일 수 있다. 반복 실행 시 결과가 역전될 수 있으므로 "가상이 더 빠르다"는 결론보다 "코어 초과 시 두 모델이 사실상 동등하다"는 것이 더 안전한 해석이다.

### CPU-bound 작업에서 가상 스레드의 적정 수는?

```
추천: concurrency = cores (= 12)
이유: 처리량 최대값에 도달하면서 스케줄링 오버헤드 최소
주의: I/O-bound와 CPU-bound 혼합 시 CPU 작업이 carrier를 장기 점유 → I/O 가상 스레드 지연 가능
```

### 실무 시사점

"가상 스레드는 CPU 작업에 쓰면 안 됩니다"는 틀린 말이 아니지만 과장된 말.  
정확한 표현: "CPU-bound 작업에서 가상 스레드는 플랫폼 스레드와 동등하다. 단, CPU-bound 작업을 위해 스레드 수를 core 수 이상으로 늘리는 것은 두 모델 모두에서 무의미하다."
